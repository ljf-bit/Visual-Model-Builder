# 已实现功能说明

## 1. 文档目的

本文档用于集中记录项目当前已经实现的功能、关键行为约束、维护注意事项，以及后续继续迭代时容易被改坏的地方。

它不是 README 的重复版本，而是给后续开发者或 AI 修改代码前优先阅读的“现状说明 + 风险提示”文档。后续只要项目功能、接口、持久化格式、训练结果结构或关键交互发生变化，都应同步更新本文件。

---

## 2. 当前阶段概览

当前项目已经完成：

- Phase 1：CNN 可视化教学版的核心闭环
- Phase 2：训练流程教学版的核心闭环
- Phase 3：训练诊断与教学解释闭环的首批能力

项目现在已经不仅能可视化搭建 CNN 模型，还能完成最小训练图构建、训练代码生成、训练执行、训练前诊断、训练结果解释和结果保存。

## 4. 当前已实现的 Phase 2 功能

当前前端已经支持以下 Phase 2 训练节点：

- `Dataset`
- `DataLoader`
- `Loss`

`Dataset -> DataLoader -> Input -> ...模型主体... -> Output -> Loss`

相关前端位置：

- `frontend/src/registry/nodeRegistry.ts`
- `frontend/src/features/palette/`
- `frontend/src/features/canvas/`

### 4.2 训练图校验

后端已经区分：

- 普通模型图校验：`/validate-graph`
- 训练图校验：`/validate-training-graph`

不要把训练图和普通模型图走同一套校验逻辑，否则会出现：

- 带训练节点时模型代码生成被错误拦截
- 训练链路缺失却没有被正确报错

相关后端位置：

- `backend/app/api/__init__.py`
- `backend/app/services/validator/`

### 4.3 模型代码与训练代码生成

当前代码生成区已经分为两种模式：

- `Model`
- `Training`

它们的缓存是分开的，切换标签会显示各自最后一次生成结果，不需要重新生成。

重要约束：

- `Model` 模式只生成模型结构代码
- `Training` 模式生成训练脚本
- 两个模式不能共用一份 `generatedCode`

相关前端位置：

- `frontend/src/features/codegen/CodePanel.tsx`
- `frontend/src/store/useAppStore.ts`

相关后端位置：

- `backend/app/services/codegen/__init__.py`
- `backend/app/services/codegen/training.py`

### 4.4 训练运行

- 构建 DataLoader

- 构建 Optimizer
- 执行训练循环

注意：

### 5.2 首批诊断规则覆盖

当前首批诊断已经覆盖以下教学高频问题：

- 输出类别数与数据集类别数疑似不一致
- `Linear.in_features` 与前序 shape 的一致性风险
- `Flatten` 非默认参数带来的单样本语义提示
- 学习率过大或过小
- batch size 不合理

- 缺少 `Metric`
- `Loss` 类型与输出形式不匹配
- `Optimizer` 关键参数的教学性提示
- 模型过小或过深时的教学提示

- 诊断摘要、错误数、警告数、建议数会展示在 `TrainingPanelV2`

前端 store 中已经新增：

- `trainingDiagnostics`

它会与 `trainingResult` 分开存储，避免“训练未运行成功时诊断信息也一起丢失”。

相关位置：

## 6. 当前实现中的关键约束

### 6.1 节点能力必须保持前后端一致

无论是 Phase 1 还是 Phase 2，只要修改节点类型、参数、默认值、语义或连接规则，都必须同时检查：

- 前端注册表
- 前端属性面板
- 后端校验器
- 后端 shape 推导
- 后端代码生成
- 后端训练运行时构建

只改其中一端，极容易出现“图能画但不能推导 / 能生成代码但不能运行 / 前端展示与后端语义不一致”的问题。

### 6.2 `Flatten` 参数语义是“单样本语义”

这是 Phase 2 中非常容易被改坏的一点。

当前图编辑器里的 `Flatten.start_dim` 是基于单样本张量理解的，不是基于带 batch 的真实运行时张量。

例如：

- 图上样本形状：`[C, H, W]`
- 运行时真实张量：`[B, C, H, W]`

因此运行时和代码生成时，`Flatten.start_dim` 会自动加 1，以跳过 batch 维度。

如果后续有人把这段逻辑删掉或改回 PyTorch 原始含义，会直接导致：

- `mat1 and mat2 shapes cannot be multiplied`
- `Linear.in_features` 与前端 shape 推导不一致

必须保持一致的位置：

- `backend/app/services/training.py`
- `backend/app/services/codegen/__init__.py`

### 6.3 训练输出需要做分类维度规范化

当前某些图会得到类似：

- `[B, 10, 1]`

在进入 `CrossEntropyLoss` 前，后端会做 trailing singleton 维度压缩。

如果后续移除这步，会出现：

- `loss` 输入维度不匹配
- 某些原本可跑的 Phase 2 教学案例再次报错

关键位置：

- `backend/app/services/training.py`
- `backend/app/services/codegen/training.py`

### 6.4 `Model` 和 `Training` 代码缓存必须隔离

后续不要再把两类代码合并成一个共享字符串状态。

必须保持：

- `generatedCodeByMode.model`
- `generatedCodeByMode.training`

关键位置：

- `frontend/src/store/useAppStore.ts`
- `frontend/src/features/codegen/CodePanel.tsx`

### 6.5 React Flow 运行时字段不能写回持久化项目

当前 store 会对节点和边做清洗，只保存项目真正需要的字段：

- 节点：`id/type/position/data`
- 边：`id/source/target/sourceHandle/targetHandle`

不要把 React Flow 的运行时字段直接存回项目，例如：

- `selected`
- `dragging`
- 内部尺寸字段
- 运行时布局字段

否则会导致：

- 撤回行为不稳定
- 复制粘贴后多个节点像“粘”在一起
- JSON 保存内容不干净

关键位置：

- `frontend/src/store/useAppStore.ts`

### 6.6 训练结果保存依赖后端元信息

前端导出 HTML 报告时，不是自己猜测训练超参数和文件路径，而是依赖后端返回的 `trainingMetadata`。

因此后续如果修改 `/run-training` 返回结构，必须同步更新：

- `backend/app/schemas/responses.py`
- `frontend/src/types/index.ts`
- `frontend/src/features/training/TrainingPanelV2.tsx`

否则会出现：

- 前端导出报告缺字段
- 路径显示为空
- 权重保存地址不可信

---

### 6.7 诊断与解释字段必须前后端同步

当前 Phase 3 新增的这些字段已经进入真实用户链路：

- `RunTrainingResponse.diagnostics`
- `RunTrainingResponse.insights`
- `TrainingDiagnosticsResponse`
- `TrainingInsightsResponse`

它们不仅影响接口返回，也影响：

- 训练前阻断逻辑
- `Analysis` 视图渲染
- HTML 报告导出
- `training_summary.json`

因此如果后续修改字段名、层级或语义，必须同步检查：

- `backend/app/schemas/responses.py`
- `backend/app/services/diagnostics.py`
- `backend/app/services/training.py`
- `frontend/src/types/index.ts`
- `frontend/src/features/training/TrainingPanelV2.tsx`

### 6.8 输出类别数推断必须遵循规范化后的 shape 语义

当前诊断与训练配置摘要里的 `outputClasses`，已经不再简单等于“最后一个 `Linear.out_features`”，而是遵循训练运行时的分类输出规范化逻辑。

例如某些合法教学图会产生：

- 样本级输出 shape：`[10, 1]`
- 运行时输出 shape：`[B, 10, 1]`
- 进入分类 loss 前会规范化为：`[B, 10]`

这类情况下，真正的类别数应该视为 `10`，而不是 `1`。如果后续把类别数推断又改回“只看最后一个 Linear”，会导致：

- 训练前被误诊断为类别数错误
- 原本能跑通的 Flatten 教学案例被错误阻断
- `Analysis` 视图和真实训练行为不一致

关键位置：

- `backend/app/services/training.py`
- `backend/app/services/diagnostics.py`
- `backend/app/services/codegen/training.py`

## 7. 版本控制与文档维护约定

### 7.1 修改代码前必须先走版本控制

后续任何 AI 对项目代码进行修改时，都必须在 Git 仓库中进行，不能绕过版本控制直接做不可追踪修改。

至少遵守以下规则：

1. 修改前先检查 `git status` 和当前分支，避免覆盖用户未说明的本地改动。
2. 每次阶段性功能实现、修复或重构完成后，必须创建独立提交，保证能回滚到该版本。
3. 回滚优先基于提交号进行，不要在未获得用户明确确认时使用破坏性命令。
4. 若当前改动涉及较大功能点，回复用户时应说明对应提交号，方便指定版本回退。

### 7.2 实现完新功能后必须更新本文件

只要新增、删除或改变以下内容，就必须同步更新本文件：

- 已实现功能
- 关键约束
- 持久化格式
- API 返回结构
- 回归检查项
- 高风险修改点

也就是说，本文件不是一次性说明，而是需要随着项目版本持续维护的“实现现状文档”。

### 7.3 建议的修改闭环

建议后续 AI 修改项目时遵循：

1. 先阅读本文件和相关源码
2. 确认当前 Git 状态
3. 实现功能
4. 运行必要验证
5. 更新本文件
6. 创建提交并返回提交号

## 8. 后续更新时的高风险点

- `frontend/src/registry/nodeRegistry.ts`

- `backend/app/services/codegen/`

### 8.2 修改保存/加载格式时

项目 JSON 当前版本是：

- `2.0.0`

保存/加载逻辑会对旧项目做兼容归一化。

如果后续调整：

- 节点字段
- 参数键名
- 训练节点结构
- `metadata`

请同步检查：

- `frontend/src/App.tsx`
- `frontend/src/graph/graphUtils.ts`

### 8.3 修改训练结果面板时

- `frontend/src/features/training/TrainingPanelV2.tsx`

- `frontend/src/features/training/TrainingPanel.tsx`

- `frontend/src/features/training/index.ts -> TrainingPanelV2`

### 8.4 修改后端训练目录时

当前训练结果默认写入：

- `backend/training_runs/`

也可以通过环境变量覆盖：

- `VMB_TRAINING_RUNS_DIR`

如果改动该目录逻辑，请确保：

- 权重文件仍然可保存
- logs JSON 和 summary JSON 路径仍然可返回前端
- Windows 路径可正常显示

每次进行阶段性更新后，至少检查以下事项：

1. 基础模型图能通过 `/validate-graph`
2. CNN 主链路 shape 推导正常
3. 模型代码可正常生成
4. 项目保存后可重新打开
5. 旧项目 JSON 打开时不会崩溃
6. 训练图能通过 `/validate-training-graph`
7. `Model` 和 `Training` 两个标签能分别显示对应代码
8. `/diagnose-training-graph` 能返回 `summary / warnings / errors / suggestions / graphStats / modelStats / trainingStats`
9. 存在致命诊断错误时，`Run Training` 会被阻止
10. 只有警告时，`Run Training` 仍可继续执行
11. `Run Training` 成功时仍可以返回 `loss` 和 `accuracy`
12. `Curves` 视图能显示曲线或点
13. `Analysis` 视图能显示配置摘要、模型摘要、趋势解释和修复建议
14. `Save Results` 导出的 HTML 报告包含诊断与解释信息
15. 后端 `training_runs` 目录能看到权重和 JSON 摘要，且 `training_summary.json` 中包含 `diagnostics` 与 `insights`
16. `Ctrl/Cmd + C / V / Z / S` 正常工作
17. 第一个节点添加后可立即撤回
18. 粘贴节点不会与已有节点完全重叠

## 10. 总结

当前项目已经不是单一的 Phase 1 原型，而是具备模型搭建与最小训练闭环的教学工具。

后续更新时，最重要的是保持以下四组一致性：

- 前端节点定义 <-> 后端校验/推导/代码生成
- 图上 shape 语义 <-> 运行时 batch 语义
- 前端展示/导出字段 <-> 后端返回字段
- 功能实现现状 <-> 本文档记录 <-> Git 提交历史

只要这几组一致性不被破坏，项目就能持续稳定迭代，并且在需要时回滚到明确版本。
